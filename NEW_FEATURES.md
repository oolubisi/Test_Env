## 4. Fixed: blank Notes shows a number after saving a new project

**Root cause:** `saveProject` (and several other save functions) wrote rows as **positional arrays** via `appendRow()`. If the live Google Sheet's column order ever drifted from `SHEET_SCHEMAS` (e.g., from earlier schema changes that appended columns), the values would land in the wrong cells — the `Date.now()` timestamp meant for `lastModified` was landing in the `notes` column instead.

**Fix:** Converted `saveProject`, `updateProject`, `saveInspection`, `saveTakeOffItem`, `saveProgressLog`, `saveVendor`, `saveWorkOrder`, and the new `saveSnag`/`updateSnag` to use `appendObjectRow()`, which maps values to columns **by header name** — immune to column-order drift, and consistent with how `savePayment` already worked.

**Files:** `code.gs`

## 5. Date format standardized to YYYY/MM/DD

All dates generated by the app (inspection dates, payment dates, progress log dates, work order creation dates, snag dates) now use `YYYY/MM/DD`:

- **Backend (`code.gs`):** `currentDate()` now returns `Utilities.formatDate(new Date(), Session.getScriptTimeZone(), "yyyy/MM/dd")`.
- **Frontend (`utils.js`):** new `todayFormatted()` helper returns the same `YYYY/MM/DD` format, used everywhere `new Date().toLocaleDateString()` was previously called (inspection and payment modals).

**Note:** This only affects *newly created* records. Existing records already saved in `dd/MM/yyyy` (or your locale's format) will keep their old format unless edited/re-saved. If you want to bulk-convert existing dates in the Sheet, that would need a one-time script — let me know if you'd like one.

**Files:** `code.gs`, `utils.js`, `modals.js`

## 6. Payment ID format changed to XXX-XX

New format: the first three digits are the **last three digits of the corresponding project number** (from `PRJ/26/001` → `001`), followed by a 2-digit sequential counter per project.

Examples:
- First payment on project `PRJ/26/001` → `001-01`
- Second payment on the same project → `001-02`
- First payment on project `PRJ/26/042` → `042-01`

**Implementation:**
- New `projectNumberSuffix(projectId)` extracts the last 1–3 digits from the project ID and pads to 3 digits.
- `getNextSequentialPaymentId(projectId)` now scans existing payment IDs with the matching `XXX-` prefix and returns the next 2-digit counter, wrapped in `LockService` to avoid collisions on concurrent syncs.
- `savePayment` passes `data.projectId` through.

**Note:** Existing payments with the old `PAY-XXXXX` format will keep their IDs; only new payments get the new `XXX-XX` format. If you'd like existing payment IDs renumbered, that's a separate one-time migration — let me know if you want help with that too.

**Files:** `code.gs`
